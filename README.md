## 3. 从代码到公网：部署步骤（循证，无臆造平台 UI 细节）

下列步骤以「**托管 PostgreSQL（如 Neon / Supabase）+ Render Web Service + 本仓库 Dockerfile**」为例；若课程指定 MySQL，需另起分支改驱动与 SQL，不在本仓库范围。

### 3.1 准备数据库

1. 在 Neon 或 Supabase 创建 **PostgreSQL** 实例，复制控制台提供的连接 URI（通常含 `postgresql://` 或 `postgres://` 用户名密码与主机）。
2. 使用任意 PostgreSQL 客户端（psql、DBeaver、托管控制台 SQL Editor）对目标库执行本仓库根目录 **`schema.sql` 全文**（含建表、视图、种子数据）。执行成功后应有表 `"User"`、`item`、`orders` 及视图 `sold_items_view`、`unsold_items_view`。

### 3.2 构建并部署应用

1. 将本仓库推送到你的 Git 托管（GitHub / GitLab 等）。
2. 在 Render 创建 **Web Service**，连接该仓库，运行时选择 **Docker**（使用根目录 `Dockerfile`）。
3. 在服务的 **Environment** 中新增变量：
   - **`DATABASE_URL`**：粘贴上一步数据库连接串（与 `app/settings.py` 中字段名一致）。
4. 保存并触发部署；待构建日志无报错后，打开平台分配的 **HTTPS URL**，应能访问首页。

### 3.3 部署后自检

- 浏览器访问：`/health` 应返回 JSON `{"status":"ok"}`。
- 浏览器访问：`/health/db` 应在数据库可达时返回 `db: 1`；若返回 503 且提示未配置 `DATABASE_URL`，检查环境变量是否注入到**运行**环境而非仅构建环境。

### 3.4 本地对照运行（可选）

```bash
cd /path/to/this/repo
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
export DATABASE_URL='postgresql://USER:PASS@HOST:5432/DBNAME'
uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
```

## 5. 第八部分（简答素材）：角色分离与安全运维（请用自己的话改写后提交）

1. **角色分离**：数据库至少两类账号——**只读账号**（`SELECT`、可选 `USAGE` on schema）供报表、排障；**应用账号**（`INSERT`/`UPDATE`/`DELETE` 及业务所需 `SELECT`）供 Web 服务。禁止给应用账号 `SUPERUSER` 或随意 `DROP` 权限。
2. **最小权限**：按表/列授予权限；生产库不用超级用户跑应用；迁移与结构变更使用独立 **migration 角色** 并在变更窗口执行。
3. **禁止直连生产写库**：开发者日常工具默认连 **只读副本** 或脱敏库；写操作只经应用与受控迁移。
4. **边界防护与审计**：入口经 **HTTPS**；在网关或云厂商侧启用 **WAF** 减缓注入与爬虫；数据库开启 **审计日志**（谁、何时、改了什么），与告警联动。

以上内容为通用最佳实践表述，非某云厂商专有名词堆砌；提交前请结合课堂讲义用语调整。

## 6. 第九部分（简答素材）：并发、事务与恢复（请用自己的话改写后提交）

1. **并发双买**：两请求同时读到 `status=0` 可能都尝试下单。对策：购买路径使用 **事务**，对商品行 **`SELECT … FOR UPDATE`** 加锁，使第二事务等待；插入订单并 `UPDATE item SET status=1` 在同一事务提交。表 **`orders.item_id` UNIQUE** 可在极端竞态下作为第二道防线，避免一单多卖。
2. **崩溃与一致性**：提交成功前崩溃应 **回滚** 未提交变更，不出现「有订单但商品仍未售」的中间态。PostgreSQL 依赖 **WAL** 保证崩溃恢复；定期 **逻辑/物理备份** 与 **PITR**（若平台支持）用于灾难恢复。若使用 MySQL，对应为 **redo/binlog** 与备份策略，课程指哪写哪。
